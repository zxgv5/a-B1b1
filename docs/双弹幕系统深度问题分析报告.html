三、两套引擎共同上游

P0更新通道满时直接丢弃，且调用方完全不知情

处理状态已解决：StateFlow 同时保存完整恢复点、最新增量、播放代际和批次序号。连续消费者只收增量；新消费者或批次不连续时自动接收完整快照。切集清空恢复点并拒绝旧视频任务回写。 触发主线程 collector 正在处理一批更新、缓冲区已有一批等待时，第三批到达。发布计算虽在单线程执行，但它不会等待主线程消费。 结果tryEmit 返回 false，整批弹幕在进入 PlayerActivity 和任一引擎前永久消失；日志仍会记录发布完成，因此排查时很容易误判为引擎丢数据。 当前机制正常连续批次按 append 进入引擎；StateFlow 若合并了中间状态，消费者会从批次序号发现断档并用最新完整快照自愈，不会静默缺段。 回归测试覆盖连续批次走增量、批次断档走快照、首次订阅走快照和明确 replace 走快照。

四、性能优先引擎

P0连续 append 会丢失前一批数据

处理状态已解决：append 不换代并等待上一 prepareJob；replace/reset 才换代。停播期间只更新 rawItems，不向清空后的引擎注入局部结果。 触发A 批追加仍在排序或预处理时，B 批追加到达。上游分段发布虽然单线程发出，但控制器收到后立即启动新的并行后台任务，因此仍可重叠。 结果A 批没有写入 rawItems，B 批只和更旧的数据合并；A 批永久消失。表现为某一小段弹幕明显稀疏，seek 回去也无法恢复。 修复方向增量任务必须串行归并，或维护待合并队列。代际取消只适用于“全量替换”，不能直接用于需要累积的 append。 回归测试让 A 在合并后、提交前挂起，再提交 B；最终时间线必须同时包含初始数据、A、B，且顺序正确。

P1增量路径实际上是全量重建，越播越重

处理状态已解决：严格时间尾追加且不会改变旧的 2 秒重复合并结果时，只过滤、合并、转换和 append 新批次；原始时间线也使用分块追加。乱序、过滤上下文变化或跨批同款冲突会自动回退全量重建。 复杂度如果分 K 批逐步加载到 N 条，总处理量接近 N×K，而不是 N。时间线越大，每次追加产生的临时列表和对象越多。 主线程本轮已去掉 append 的万级防御性复制。剩余主要问题转为后台 CPU、临时对象和 Action 线程全量重建。 视觉影响每次全量替换都会清空当前 active、轨道和缓存引用，再按当前位置重建，容易造成短暂空帧、轨道重排和缓存命中率骤降。 修复方向预处理只处理新批次；把“显示去重”和“历史去重”设计成可增量状态；引擎使用已有的 appendDanmakus，同时给每次播放数据一个唯一批次标识，避免旧版全量误追加问题复发。

P2数据级设置会同步卡住主线程

处理状态已解决：设置重建与 set/append 共用后台串行准备链，等待在途数据提交后再按最新设置处理；主线程只提交最终结果。 原结果设置面板操作会随已加载弹幕数量变慢。长视频或高密度视频上，点击开关可能直接丢帧。

P2缓存任务缺少有效性校验，退场后仍能回写

处理状态已解决：缓存线程只生成 CacheBuildResult，回到 Action 线程后再核对样式代际、待提交代际和 active 身份。失效结果不挂回 item，共享 Bitmap 用原子引用申请防止“检查后被回收”。 根因缓存请求带了 releaseAtFrameId，但 CacheManager.kt:201-327 构建前没有校验条目是否仍 active、缓存代际是否仍是当前值，这个参数本身也从未用于构建结果提交。与此同时，DanmakuEngine.kt:654-665 允许条目在任务完成前退场并重置。 结果慢设备或缓存队列拥堵时，旧任务可把 Bitmap 重新挂到已退场条目。该条目不再经过正常退场释放流程，Bitmap 会滞留到条目再次入场退场、全量替换或播放器释放，同时产生无效构建。主绘制路径有缓存代际检查，因此不能据此断言会画出旧样式。 修复方向请求携带 item token + 样式代际；构建前和提交前各校验一次。失效结果直接回池，不写回 item。退场时递增 token。

五、功能优先引擎

P0任务虽然串行等待，连续增量仍会丢前一批

处理状态已解决：append 继承上一任务代际并等待其提交；设置重建也加入同一任务链，不再取消尚未落 timeline 的增量。 触发A 批 append 尚未提交时 B 批到达；更严重的是，首次 set 尚未提交时紧跟 append，也会让完整 set 失效，最终时间线可能只剩 append 批次。 结果等待并没有保住 A，只保证 B 在 A 退出后计算。A 不会进入后续时间线，seek 也无法恢复。 修复方向append 不应让正在执行的累积任务失效。用单消费者队列串行处理 set/append，set 可清空前序队列，append 必须基于上一操作的提交结果继续归并。 回归测试覆盖 set 未完成时 append、A append 未完成时 B append 两种时序；最终时间线不得丢 set、A 或 B。

P1字号变化后复用旧测量结果

处理状态核心问题已解决：测量表只允许当前 measureGeneration 读写；切换到新代际时常数时间换表，旧尺寸不能跨代命中；同时用 8192 条 LRU 限制长期占用。 剩余边界测量完成后已再次核对当前任务和配置代际，可拦截绝大多数迟到任务；但检查与条目状态写入之间仍有极窄窗口。彻底解决需要让运行时换代重置与缓存提交共用锁、CAS 或同一线程队列。 原结果调大字号后，轨道仍可能按旧宽高判断碰撞，导致追尾、重叠或裁切；调小后则可能留出不必要的大间隔。绘制 Bitmap 可以是新字号，但布局尺寸仍是旧值，所以问题不一定表现为“字体没变”。 回归测试同一编号在旧代际写入宽高，切换新代际后必须 miss；新代际写入后，迟到的旧任务不得覆盖；超过容量时淘汰最旧结果。

P2主线程存在等待后台整帧更新和缓存绘制的路径

第一把锁DanmakuRuntime.kt:250-255 的 Action 更新和 :370-375 的主线程绘制都锁住整个 Runtime；主线程可能等待一整次布局、测量调度、帧构建和旧帧释放。 第二把锁DanmakuContext.kt:35,60-76 让后台测量/Bitmap 构建与主线程 fallback 直绘共用同一个 rendererLock。缓存线程正在画一条长弹幕时，主线程 fallback 会被阻塞。 处理状态已解决主路径：日志中两次绘制峰值 17ms 和 31ms 有 13ms 和 22ms 在等 Runtime 锁，长样本最高等待 68ms。现在主线程通过读租约使用已发布帧，旧帧只在读者离开后回收；Runtime 锁不再包围 draw。 修复方向Runtime 改为不可变帧快照交换，主线程只读当前帧，不与 Action 共用大锁。渲染器应拆分线程私有 Paint/状态，让缓存绘制与 fallback 直绘不争同一把锁。 后续验收使用真机重跑同一高密度片段，确认旧的 draw syncWait 已消失，绘制 P99 和视频丢帧同时下降。模拟器 VSync 异常不作为最终结论。

代码完成两套引擎的长播 Bitmap 已接入统一字节预算

测量表measureSizeCache 已改为按代际清理、最多保留 8192 条的 LRU，不再随唯一弹幕编号无限增长。 Bitmap代码已将活动弹幕、共享表、渲染帧、待释放队列和回收池接入引擎中立的对象身份预算器；同一文字绘制 Bitmap 只计一次。两套引擎统一按分辨率使用 32/50/72MB 上限，低内存设备减半。VIP 贴图继续由独立的 2MB LRU 管理。 超限策略新建前先做预算准入，优先销毁池内空闲对象，再释放共享 LRU；仍被画面引用的 Bitmap 不会被强制回收，空间不足时跳过本次缓存构建。 观测性能日志新增 Bitmap 总字节、总上限和对象数。单元测试已覆盖预算器身份去重、预算拒绝、预留转交、淘汰顺序和退出时延迟引用排空；尚未用 JVM 单测构造两套 Android Bitmap 管理器的完整活动/共享/池生命周期，因此“release 后总量归零”仍需 2 小时真机压力确认。

  • 功能优先控制器 append 的全量排序已移到后台，并改为线性归并。

共同约束：主线程不等待布局、测量和缓存构建；弹幕容纳不下或缓存未就绪时可以丢弃本条/本帧，不得为了“尽量显示”把工作转移到主线程；任何异步结果都必须带播放代际、配置代际和条目标识。
